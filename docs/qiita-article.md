## AIで「いい感じに作って」の先にあるもの

「ログイン機能を作って」「ダッシュボードを追加して」——AIに指示を出せば、それなりに動くコードが出てくる時代になった。バイブコーディング、最高に楽しい。

でも、ふと不安になる瞬間がある。

「これ、来週の自分に説明できるか？」

小さいうちは問題ない。コードを読めばなんとなく分かる。ところがシステムが育ってくると、話が変わる。機能追加を頼んだら既存機能が壊れた。修正したら別の画面がおかしくなった。AIに「なんでこの仕様にしたんだっけ？」と聞いても、当然覚えていない。

これ、バイブコーディングあるあるだと思う。

## 「資料なんていらない」vs「ちゃんと書け」

この問題に対して、大きく二つの立場がある。

**不要派**はこう言う。
- コードが仕様書だ
- AIに聞けば分かる
- 動いてるなら正義

気持ちは分かる。実際、小規模なプロジェクトならこれで回る。

**必要派**はこう言う。
- 仕様書がないと破綻する
- AIは過去のコンテキストを忘れる
- チームで開発するなら引き継ぎどうするの

これも正しい。規模が大きくなると、何かしらの記録がないと詰む。

でも、ここで矛盾に気づく。**AIで楽をしたいのに、AIのために人間が資料を書くのか？** それは本末転倒じゃないか。仕様書を書く時間があったら、その分AIにコードを書かせたい。

どっちの立場にも一理ある。でもどっちも辛い。

## 第三の選択肢——AIが書いてAIが読む

ここで発想を変えてみる。

資料が不要なのではない。**人間が書く必要がなくなった**だけだ。

AIに「この機能の要求を整理して」と頼めば、AIが要求を書く。「機能に落としこんで」と言えば、機能定義を書く。「仕様を詳細化して」と言えば、仕様書を書く。人間がやるのは**レビューと意思決定だけ**。

こうして作られた資料を、次にAIがコードを書くときに参照する。「この仕様に基づいて実装して」と言えば、AIは自分が書いた仕様を読んで、一貫性のあるコードを書ける。

さらに、変更が入ったときも「なぜこの変更が必要か」をAIが記録しておく。3ヶ月後に「なんでこうなった？」と思ったら、変更履歴を辿ればいい。

つまり、**要求→機能→仕様**の階層で整理して、AIが書き、AIが読み、AIが更新する。人間はその流れを眺めて「OK」か「違う」かを判断するだけ。

この仕組みを実現するOSSを作った。**VibeShift**という。

## VibeShiftで何が変わるか

VibeShiftは、AIエージェントがMCP（Model Context Protocol）経由で直接操作する要件管理システムだ。

### AIとの対話で要件が育つ

使い方はシンプル。AIエージェントにこう言うだけ。

```
「このプロジェクトの要求を洗い出して」
```

AIが要求ノードを提案してくる。内容を確認して「OK」と言えば作成される。続けて「機能に分解して」「仕様を詳細化して」と対話を進めると、グラフがどんどん育っていく。

### グラフで全体像を把握

作成されたノードはWeb UIでグラフとして表示される。「要求→機能→仕様」の階層構造が一目で分かる。ノードをクリックすれば、マークダウンで書かれた詳細と変更履歴が確認できる。

### 変更理由が自動で残る

ノードを更新するとき、AIが「なぜこの変更が必要か」を自動記録する。後から「この仕様、なんで変わったんだっけ？」と思ったら、変更履歴を見ればいい。人間がわざわざ議事録を書く必要はない。

### コード変更の影響を教えてくれる

コードを変更したとき、「この変更で影響を受ける仕様はあるか？」をAIに聞ける。VibeShiftが仕様ノードを検索して、影響範囲を教えてくれる。仕様の更新漏れを防げる。

### 記述ルールでAIの書き方を統一

AIエージェントごとに書き方がバラバラになる問題にも対応した。プロジェクトに「記述ルール」を設定すると、どのAIエージェントが書いても同じフォーマットで統一される。Claude CodeでもClaude Desktopでも、出力が揃う。

## 3分で試せる

### 前提条件

- Node.js 20+
- pnpm 9+

### セットアップ

```bash
git clone https://github.com/skspwork/vibe-shift.git
cd vibe-shift
pnpm install
pnpm build
pnpm dev
```

これでWeb UI（http://localhost:3000）とAPI（http://localhost:3001）が起動する。

### MCPサーバーの接続

Claude Codeを使っている場合、プロジェクトの`.mcp.json`に以下を追加する。パスはVibeShiftをクローンした場所に合わせて変更してほしい。

```json
{
  "mcpServers": {
    "vibeshift": {
      "command": "npx",
      "args": ["tsx", "/path/to/vibe-shift/packages/mcp-server/src/index.ts"]
    }
  }
}
```

Claude Desktopの場合は`claude_desktop_config.json`に同様の設定を追加する。

### 最初の一手

MCPサーバーを接続したら、AIエージェントにこう言ってみてほしい。

```
VibeShiftに新しいプロジェクトを作って。
名前は「○○」で、目的は「○○」。
要求を洗い出すところから始めよう。
```

AIが対話を通じて要求を提案し、承認すればノードが作られていく。Web UIを開けば、グラフが育っていく様子を確認できる。

## バイブコーディングを"持続可能"にする

バイブコーディングの問題は、スピードと引き換えに文脈が失われることだった。

VibeShiftは「人間が資料を書く」のではなく「AIが資料を管理する」ことで、この問題を解決する。人間は意思決定に集中できて、AIは一貫性のあるコンテキストを持ち続けられる。
