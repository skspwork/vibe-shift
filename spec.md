# CddAI — AIドリブン開発トレーサビリティ管理システム 機能仕様書 v1.3

---

## 1. プロダクト概要

CddAIは、AIとの会話を起点として、要求・要件・仕様・設計・実装までのトレーサビリティをグラフィカルに管理するWebアプリケーションである。ユーザー自身によるノードの手動作成・編集と、AIとの対話によるノード生成の両方をサポートする。ローカルで動作するAI AgentがMCP経由で直接操作することもできる。

### 1.1 解決する課題

- 要件定義の根拠（なぜその要件が生まれたか）が失われる
- 会話の中で決まった内容と正式な要件の対応関係が不明確になる
- 実装コードがどの要件を満たしているか追跡できない
- 要件変更時の影響範囲が把握しづらい

### 1.2 コアコンセプト

グラフの頂点には「システム概要ノード（overview）」が1つ存在し、プロジェクト全体の目的・背景を保持する。すべてのノードはこのルートを起点とするツリー構造を形成する。

各ノードはユーザーが手動で作成・編集することも、AIとのセッションを通じて生成することもできる。**会話ノード（conv）はグラフ上には表示せず**、各ノードの詳細パネルから「生成経緯」として参照する仕組みとする。

**グラフ上の見た目**
```
overview
  └─ need ── req ── spec ── design ── task
```

**内部データ構造（conv は保持・非表示）**
```
overview
  └─ [conv: 認証について]
       └─ need ── [conv: 要件を深掘り]
                    └─ req ── [conv: 仕様を詰めた]
                                └─ spec
```

## 2. ノード階層設計

### 2.1 ノード種別

| ノード種別 | 英語名 | グラフ表示 | 説明 |
|-----------|--------|:---------:|------|
| システム概要 (overview) | Overview | ✓ | プロジェクトのルートノード。1プロジェクトに1つ |
| 会話 (conv) | Conversation | **非表示** | AIセッションの記録。各ノードの経緯として内部保持 |
| 要求 (need) | Need | ✓ | 「〜したい」というステークホルダーの声 |
| 要件 (req) | Requirement | ✓ | 「システムは〜しなければならない」という形式的な要件 |
| 仕様 (spec) | Specification | ✓ | 具体的な振る舞い・制約・ルール |
| 設計 (design) | Design | ✓ | アーキテクチャ・コンポーネント構成 |
| タスク (task) | Task | ✓ | 実装チケット |
| コード (code) | Code/PR | ✓ | GitリポジトリのPR・コミットへのリンク |
| テスト (test) | Test | ✓ | テストケース・実行結果 |

### 2.2 ノードの作成方法

`overview` 以外のすべてのノード（`conv` 含む）は以下の2通りで作成される。

| 方法 | conv の扱い | 経緯欄 |
|-----|------------|--------|
| **手動作成** | 生成されない | ユーザーが自由テキストで手動入力・編集可能 |
| **AI生成** | セッション開始時に自動生成 | 対応する `conv` の会話ログが自動でリンクされる |

### 2.3 スキップ可能なレーン

プロジェクト設定でアクティブなレーンを選択できる。小規模プロジェクトでは「要求 → タスク」のみ使う構成も可能。

## 3. 経緯（Rationale）の設計

すべてのノードは「経緯」フィールドを持つ。経緯はノードがなぜ生まれたかを記録するためのものであり、作成方法によって内容の持ち方が異なる。

### 3.1 AI生成ノードの経緯

AIセッションで生成されたノードには、対応する `conv` ノードが自動でリンクされる。詳細パネルの「生成経緯」欄にはその会話ログのサマリーと全文リンクが表示される。

```
── 生成経緯 ──────────────────
💬 AIセッション「ログイン要件を深掘り」
> ユーザー: メアドとPWでのログインを...
> AI: 以下の要件を提案します...
[全文を見る →]
──────────────────────────────
```

ユーザーはこの経緯欄に**補足テキストを追記**することができる（会話ログ自体は編集不可）。

### 3.2 手動作成ノードの経緯

手動で作成したノードの経緯欄は空で始まり、ユーザーが自由テキストで記述・編集できる。

```
── 生成経緯 ──────────────────
✏️ [手動で記述する]

  例: 「2025-01-15 の打ち合わせで
      Aさんから提案。セキュリティ要件の
      観点から追加した。」

[編集]
──────────────────────────────
```

### 3.3 経緯フィールドの仕様まとめ

| 項目 | AI生成ノード | 手動作成ノード |
|-----|:-----------:|:------------:|
| conv リンク | 自動セット | なし |
| 会話ログの表示 | ✓（読み取り専用） | — |
| 補足テキストの追記 | ✓ | ✓ |
| 自由テキストの編集 | 補足部分のみ | 全体 |

## 4. ノード操作設計

### 4.1 ノード選択時のアクションメニュー

グラフ上でノードをクリックすると右パネルにノード詳細とアクションが表示される。

```
┌─────────────────────────────┐
│  [need] ログイン機能          │
│  ──────────────────────────  │
│  ユーザーがメール+PWで         │
│  ログインしたい               │
│                              │
│  親ノード: overview          │
│                              │
│  ── 生成経緯 ──              │
│  💬 AIセッション「認証について」│
│  > ユーザー: ログインしたい... │
│  [全文を見る →]              │
│                              │
│  ─── アクション ───           │
│  [✏️ このノードを編集]         │
│  [+ 子ノードを手動で作成]      │
│  [💬 AIとセッションを開始]     │
│  [🗑️ このノードを削除]         │
└─────────────────────────────┘
```

### 4.2 子ノードの手動作成

「子ノードを手動で作成」を選択するとフォームが開く。

**入力項目**
- 種別（親ノードの直下レベルに限定）
- タイトル
- 内容（自由テキスト）
- 経緯（任意。後から編集も可能）

### 4.3 ノードの編集

「このノードを編集」を選択すると以下を編集できる。

| フィールド | 編集可否 |
|-----------|:-------:|
| タイトル | ✓ |
| 内容 | ✓ |
| 経緯（手動作成ノード） | ✓ |
| 経緯の補足（AI生成ノード） | ✓ |
| 経緯の会話ログ（AI生成ノード） | 読み取り専用 |

### 4.4 ノードの削除

「このノードを削除」を選択すると確認ダイアログが表示される。

**削除ルール**
- `overview` ノードは削除不可（プロジェクトのルートであるため）
- 子ノードを持つノードを削除する場合、子孫ノードも含めてカスケード削除する
- 削除前に「このノードと子孫ノード N 件が削除されます」と確認する
- 関連するエッジ・`conv` ノード・会話ログもすべて削除する

## 5. セッション設計

AIとのセッションはどのノードからでも開始できる。セッション開始時に必ず `conv` ノードが1つ生成され、そのセッションで作成されるノードはすべて `conv` の子として紐付けられる。`conv` はグラフ上には表示されないが、内部では保持される。

### 5.1 セッションのフロー（共通）

1. ノードを選択し「AIとセッションを開始」を押す
2. チャットパネルがそのノードのセッションに切り替わる
3. `conv` ノードを自動生成し、選択ノードの子として内部に作成する
4. 以下をコンテキストとしてAIに渡す（セクション5.2参照）
5. AIが子ノードをJSON形式で返し、`conv` の子として作成する
6. 作成されたノードにはこの `conv` が経緯として自動リンクされる
7. AIはノード登録後に深掘り質問を行い、段階的にノードを育てる

### 5.2 AIコンテキスト構成

```
[システム概要]
  overview ノードのタイトル・内容

[上流コンテキスト]（セッション対象ノードの全親ノード内容）
  [need] ログイン機能: ユーザーがメール+PWで...
  ...（省略：overview から対象ノードまでを列挙）

[対象ノード]
  種別: need
  タイトル: ログイン機能
  内容: ユーザーがメール+PWでログインしたい

[指示]
  対象ノードの直下の子ノード（req）を提案し、JSON形式で返すこと
```

`overview` のセッション（新規会話セッション）では上流コンテキストは空になる。

### 5.3 セッションのコンテキスト要素

| コンテキスト要素 | overview セッション | ノードセッション |
|----------------|:-----------------:|:--------------:|
| システム概要（overview の内容） | ✓ | ✓ |
| 選択ノードの全親ノード内容 | — | ✓ |
| 選択ノード自身の内容 | — | ✓ |
| 現在の会話内容 | ✓ | ✓ |

## 6. システムアーキテクチャ

### 6.1 全体構成

```
ユーザー
  │
  ├─→ Web UI (Next.js)  ─────────────────────────→ Backend API (Node.js)
  │        │                                               │
  │        └─ トレーサビリティグラフ表示                     └─ PostgreSQL / SQLite
  │
  └─→ AI Agent (Claude Desktop等)
           │
           └─ MCP (Stdio) ──→ MCPサーバー (TypeScript) ──→ Backend API
```

### 6.2 技術選定

#### フロントエンド

| カテゴリ | 採用技術 | バージョン | 選定理由 |
|---------|---------|:---------:|---------|
| フレームワーク | **Next.js** | 15.x | App Router による柔軟なレイアウト管理。SSRは不要なためほぼSPAとして使用 |
| UIライブラリ | **React** | 19.x | Next.jsに同梱 |
| 言語 | **TypeScript** | 5.x | 型安全性。ノード種別・エッジ型の厳密な定義に有効 |
| スタイリング | **Tailwind CSS** | 4.x | **必須**。ユーティリティクラスによる高速なUI構築 |
| グラフ描画 | **@xyflow/react** | 12.x | **必須**。React Flow の最新パッケージ名。カスタムノード・エッジの実装が容易 |
| 状態管理 | **Zustand** | 5.x | 軽量。グラフのノード・エッジ状態、セッション状態の管理に適する |
| データフェッチ | **TanStack Query** | 5.x | キャッシュ・楽観的更新。ノード作成時のグラフ即時反映に有効 |
| フォーム | **React Hook Form** | 7.x | ノード作成・編集フォームのバリデーション |
| アイコン | **Lucide React** | latest | Tailwind との相性が良い軽量アイコンセット |

#### バックエンド

| カテゴリ | 採用技術 | バージョン | 選定理由 |
|---------|---------|:---------:|---------|
| ランタイム | **Node.js** | 22.x (LTS) | MCPサーバーと言語統一。TypeScriptをそのまま使用 |
| フレームワーク | **Hono** | 4.x | Fastifyより軽量・型安全。Edge対応も視野に入れられる |
| ORM | **Drizzle ORM** | 0.x | TypeScript ファースト。SQLiteとPostgreSQL両対応。マイグレーション管理が容易 |
| バリデーション | **Zod** | 3.x | フロント・バック共有のスキーマ定義が可能。API入力検証に使用 |
| AI API クライアント | **Anthropic SDK** | 0.x | WebチャットパネルからClaude APIを呼ぶ `POST /chat` エンドポイント専用。MCP経由のローカルAI Agent連携では使用しない（AgentがすでにClaudeであるため、MCPサーバーはCRUDツールを提供するだけでよい） |

#### データベース

| 環境 | 採用技術 | 選定理由 |
|-----|---------|---------|
| ローカル開発 | **SQLite** (via better-sqlite3) | ファイルベースで環境構築不要。Drizzle ORM で本番と同一スキーマを使用 |
| 本番 | **PostgreSQL** 16.x | 将来のマルチユーザー・大規模グラフクエリに対応 |

#### MCPサーバー

| カテゴリ | 採用技術 | バージョン | 選定理由 |
|---------|---------|:---------:|---------|
| SDK | **@modelcontextprotocol/sdk** | 1.x | TypeScript 公式 MCP SDK。Stdio トランスポートを使用 |
| 言語 | **TypeScript** | 5.x | バックエンドと共通の型定義を `packages/shared` で共有 |

#### 開発ツール・インフラ

| カテゴリ | 採用技術 | 選定理由 |
|---------|---------|---------|
| モノレポ管理 | **pnpm workspaces** | `apps/web`・`apps/api`・`packages/mcp-server`・`packages/shared` を一元管理 |
| ビルドツール | **Turborepo** | モノレポのビルドキャッシュ・タスク並列実行 |
| Linter | **Biome** | ESLint + Prettier の代替。高速・設定が簡潔 |
| テスト | **Vitest** | Vite ベースで高速。フロント・バック共通で使用 |
| E2Eテスト | **Playwright** | ブラウザ操作の自動化。グラフ操作のE2Eに使用（Phase 4以降） |

#### パッケージ構成

```
Cddai/
├── apps/
│   ├── web/          # Next.js + Tailwind CSS + @xyflow/react
│   └── api/          # Hono + Drizzle ORM
├── packages/
│   ├── mcp-server/   # @modelcontextprotocol/sdk
│   └── shared/       # Zod スキーマ・型定義（web / api / mcp-server で共有）
├── turbo.json
└── package.json      # pnpm workspaces
```

`packages/shared` にノード種別・エッジ型・APIリクエスト/レスポンスの Zod スキーマを集約し、フロントエンド・バックエンド・MCPサーバーで型を共有することで型不整合を防ぐ。

### 6.3 データモデル

**projectsテーブル**
```
id              UUID PRIMARY KEY
name            VARCHAR(100)
active_lanes    TEXT[]
created_at      TIMESTAMP
```

**nodesテーブル**
```
id              UUID PRIMARY KEY
project_id      UUID REFERENCES projects(id)
type            ENUM(overview, conv, need, req, spec, design, task, code, test)
title           VARCHAR(100)
content         TEXT
rationale_note  TEXT        -- 経緯の自由テキスト（手動作成は全体、AI生成は補足部分）
created_by      ENUM(user, ai)
created_at      TIMESTAMP
updated_at      TIMESTAMP
```

**edgesテーブル**
```
id              UUID PRIMARY KEY
from_node_id    UUID REFERENCES nodes(id)
to_node_id      UUID REFERENCES nodes(id)
link_type       ENUM(derives, references, tests)
created_at      TIMESTAMP
```

**conv と生成ノードの紐付けはエッジで表現する。**グラフ表示時は `conv` タイプのノードをフィルタリングして非表示とし、`conv` を介したエッジは隣接ノード間の直接エッジとして描画する。

## 7. MCPサーバー仕様

### 7.1 提供ツール一覧

| ツール名 | パラメータ | 説明 |
|---------|-----------|------|
| `create_conversation` | project_id, parent_id, summary | conv ノード作成 |
| `create_node` | project_id, type, title, content, parent_id, rationale_note | ノード作成 |

### 8.6 フォーカスモード

- ノードをクリックするとフォーカスモードに入る
- 上流・下流ノードとエッジのみハイライト
- 関係のないノード・エッジは `opacity: 0.3` にグレーアウト
- 背景クリックまたは「フォーカス解除」で全表示に戻る

### 8.7 表示粒度の切り替え

| ビュー | 表示対象 | 推奨ノード数 |
|-------|---------|------------|
| フロービュー | 全レーン・全ノード | 〜30ノード |
| サマリービュー | overview・要求・要件のみ | 30〜100ノード |
| マトリクスビュー | 要件 × テストのカバレッジ表 | 要件10件以上 |

### 8.8 AIチャットパネルの動作

**overview セッション時**（新規会話）
- パネル上部に「新規会話」と表示
- 送信すると `conv` ノードが `overview` の子として内部生成され、セクション5.1のフローで動作する

**ノードセッション時**
- パネル上部に「{種別}: {タイトル} のセッション」と表示
- 「セッションを終了」ボタンで新規会話モードに戻る
- 作成できるノードは選択ノードの直下レベルに限定する

**AIへのJSONレスポンス形式**
```json
{
  "nodes": [
    { "type": "need", "title": "10文字以内", "description": "詳細説明" }
  ],
  "message": "登録確認 + 深掘り質問1〜2個"
}
```

## 9. REST API仕様

### 9.1 エンドポイント一覧

| メソッド | パス | 説明 |
|---------|------|------|
| GET | /projects | プロジェクト一覧 |
| POST | /projects | プロジェクト作成（overview ノードも同時生成） |
| GET | /projects/:id/graph | グラフ取得（conv 除外オプション付き） |
| POST | /nodes | ノード作成 |
| GET | /nodes/:id | ノード取得 |
| PATCH | /nodes/:id | ノード更新（title, content, rationale_note） |
| DELETE | /nodes/:id | ノード削除 |
| GET | /nodes/:id/conv | ノードに紐付く conv と会話ログ取得 |
| POST | /edges | エッジ作成 |
| DELETE | /edges/:id | エッジ削除 |
| GET | /nodes/:id/trace | 上流・下流ノード取得 |
| GET | /nodes/:id/context | 全親ノードをコンテキスト形式で取得 |
| GET | /nodes/search | テキスト検索 |
| POST | /chat | AI会話（JSON返答） |

### 9.2 グラフ取得オプション

```
GET /projects/:id/graph?include_conv=false
```

`include_conv=false`（デフォルト）のとき、`conv` ノードを除外し `conv` を介したエッジを親子間の直接エッジとして返す。`include_conv=true` で `conv` を含む完全なグラフを返す。

### 9.3 `/chat` POST リクエスト

```json
{
  "project_id": "uuid-xxx",
  "message": "ユーザーの発言",
  "session_type": "overview | node_session",
  "node_id": "uuid-yyy"
}
```
